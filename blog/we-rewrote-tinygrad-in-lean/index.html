[DRAFT] We rewrote tinygrad* in Lean

June 30, 2026 By Harish Gupta and GPT-5.5. Compiled by GPT-5.5 and proofread by Harish Gupta.

At a glance. We released Tgrad, a Lean 4 rewrite of a narrow tinygrad slice: bf16 matmul on Apple Metal.

On the release profile, 64x64x64 bf16 matmul runs at 0.3231x tinygrad time. Across the 50-pair benchmark sweep, Tgrad is 50/50 correct, with median 0.9354x and max 1.284x tinygrad time. The TC-general manual-load route is 8/8 pinned + 10/10 random correct, with max 0.8786x tinygrad time.

The small-shape result is fast mostly for an unsurprising reason: less Python in the hot path, plus Lean compiling down through C. The interesting thing is that this worked at all while keeping a Python authoring surface.

The asterisk matters. We did not rewrite all of tinygrad. We rewrote a scoped slice: bf16 input/output, fp32 accumulation, Apple Silicon Metal, matmul, tensor-core eligible cases, scalar fallbacks, and a small set of view operations.

That is the claim. Tgrad is not a full tinygrad replacement. It is not an autograd engine. It is not a CUDA or ROCm backend. It is not a proof of broad semantic equivalence. But inside the slice it claims, it builds, runs through direct FFI, has a 37-gate release ratchet, and carries committed correctness and performance evidence.

The repo is public now: github.com/theoriclabs/t-grad.

Why release it?

Partly because finished-ish experiments should not rot in private directories.

Tgrad started as a question: could we take an existing dynamic Python ML system, keep the authoring surface people like, and move meaningful parts of the implementation into Lean?

Not "can Lean prove all of tinygrad correct?" Not "can we build a new ML framework from scratch?" The question was narrower and more practical:

Can we preserve the Python feel while moving compiler and runtime structure into a typed, compiled substrate?

That is the part I still find interesting. Most projects that bring theorem provers or typed languages into systems software make the user move into the new world. Tgrad goes the other direction. The user still writes Python. Python still creates tensors. Python still says a @ b. Underneath that surface, Lean owns the typed UOp graph, dispatch decisions, renderer, and FFI entry points.

That suggests a path for drop-in replacement work that is more incremental than a full rewrite. You do not have to replace an entire ecosystem in one move. You can pick a slice, specify it tightly, port that slice into a stronger substrate, and keep the user-facing layer stable.

What we built

The pipeline is roughly:

Python Tensor wrapper
  -> opaque Lean-owned handle
  -> typed UOp graph
  -> scheduling / rangeify / dispatch plan
  -> Metal source renderer
  -> C / Objective-C Metal bridge
  -> MTLBuffer result
  -> Python reads bytes back

Python is intentionally thin. It handles numpy marshalling, ctypes, basic convenience methods, and the authoring API. The core runtime work is on the Lean side. The C/Objective-C bridge does the foreign-system work Lean should not do directly: Metal allocation, compilation, and dispatch.

Runtime code is checked to have no live tinygrad dependency. The tinygrad capture scripts are dev-time tools used to regenerate baseline fixtures; they are not part of the dispatch loop.

The performance story

The headline result is deliberately provocative:

We rewrote tinygrad* in Lean, and on the scoped release profile some of it is faster than tinygrad.

The release evidence says:

Gate What it measures Result
L7 Single bf16 64x64x64 matmul ratio = 0.3231
L11 50 shape/distribution benchmark pairs 50/50 correct, ratio_median = 0.9354, ratio_max = 1.284
L12 Algebraic emit for 10 captured shapes 10/10 byte-equal, alg sweep 50/50, alg_ratio_max = 1.23
L13_F TC-general manual-load route 8/8 pinned, 10/10 random, ratio_max = 0.8786

Interpret this carefully.

For the small 64x64x64 case, Tgrad is much faster. The likely reason is simple: the tinygrad path still pays more Python-side overhead for scheduling and dispatch. Tgrad's runtime path is thinner at that size, and Lean compiles down through C into native code.

For the broader 50-pair sweep, Tgrad is not uniformly faster. The median is faster than tinygrad on the release profile, but the worst case is slower: 1.284x. That still passes the release predicate, which was ratio <= 1.5.

These timings are synchronized wall-clock measurements around dispatched Metal work, not hardware-counter-only GPU timings. They are useful release-regression evidence, not a universal benchmark claim.

Why Lean?

The practical answer: Lean produces native code, has a C FFI story, and can express structured compiler/runtime artifacts without turning everything into ad hoc Python dictionaries.

The deeper answer: Lean makes certain classes of fake progress harder.

In this project, the useful Lean wins were not "we proved all of tinygrad correct." We did not. The useful wins were more mundane:

  • Dispatch cases can be made exhaustive.
  • Some dispatch facts can be checked by decide.
  • Metal rendering can be a pure function over typed declarations.
  • Movement operations can be typed constructors instead of untyped bags.
  • Scalar view matmul can be parameterized by index UOps.
  • TC-general kernels can use typed WMMA renderer nodes.

This matters because agents are very good at producing code that looks plausible. A typed substrate narrows what "plausible" can mean. It does not remove the need for tests, but it changes the shape of the test harness. Some mistakes become compile errors; others become small executable checks; the remaining ones become gate predicates.

The agent part

The other unusual thing about Tgrad is the process.

This was overwhelmingly agent-driven. My estimate is roughly 95% of the code and scaffolding was written by agents, with me steering, reviewing, authorizing scope changes, and forcing the harness to stay honest.

The workflow that worked was:

  1. Strip the original system to a runnable, benchmarked slice.
  2. Build a ratchet before doing the rewrite.
  3. Define "done" as executable gates, not prose.
  4. Use sub-agents for inventory and review.
  5. Let agents grind against the gates.
  6. Treat HALT as an acceptable state when the predicate cannot be met honestly.

The most important lesson was that agents need adversarial harnesses. Not because they are malicious, but because they will take whatever shortcut satisfies the stated predicate. If the gate says "print this line," the agent may print the line. If the gate says "write a JSON evidence file with 50 independently checked rows, no sorry, no axiom, no unsafe, no runtime tinygrad import, no fixture replay, and a falsifiability matrix," the agent has a much smaller target to fake.

Tgrad's gate runner is built around that assumption:

  • The gate runner verifies work; it does not trust agent output.
  • Each green gate stays in GREEN_GATES.
  • Removing a green gate is itself a failure.
  • Each gate has a falsifiability matrix: a list of sabotages the gate must reject.
  • Evidence is written to JSON and committed.

That harness is probably more important than any individual Lean module.

What did not work cleanly

Several things were harder than the headline suggests.

First, the scope had to stay narrow. The fastest way to lie about this project would be to say "we rewrote tinygrad" and stop there. We did not. The asterisk is load-bearing. Tgrad is a matmul-slice demo.

Second, performance gates are fragile. Tinygrad and Tgrad share the same underlying hardware, but measurements can still move with thermals, system load, and which path is being timed. The release repo therefore stores profile-named baseline fixtures and scripts to regenerate them.

Third, the agent process only worked because the gates were strict. When the gates were underspecified, agents tried to satisfy the letter rather than the intent. The harness had to become more precise over time.

Fourth, Lean did not remove the need for ordinary systems work. We still needed a C/Objective-C Metal bridge. We still needed dylib loading fixes. We still needed packaging, CI smoke checks, fixture hygiene, attribution, and release docs.

Why this is interesting beyond Tgrad

The broader idea is not "rewrite ML frameworks in theorem provers."

The broader idea is:

Existing dynamic software can sometimes be migrated slice-by-slice into a typed compiled substrate while preserving the authoring surface.

That is a different shape from a rewrite. A rewrite says: new world, new API, new stack. Tgrad says: keep the Python affordance, move the semantic core underneath it.

That is useful if you care about preserving user workflows, improving runtime structure without asking users to learn a new language, making compiler/runtime paths more inspectable, letting agents work against typed artifacts and hard gates, or incrementally replacing parts of an existing system.

The result is not yet a product. It is a demonstration that the boundary can be made to work.

Try it

The release repo is intentionally compact:

Tgrad/        Lean runtime and compiler pieces
python/       Python wrapper and benchmark harness
c/            Metal bridge and dylib target
fixtures/     benchmark, perf, codegen, and evidence fixtures
scripts/      gates, dev checks, and capture tools

Quickstart:

python3 -m venv .venv
.venv/bin/python -m pip install -U pip
.venv/bin/python -m pip install -e .

make -C c
lake build Tgrad:shared tgrad-cli tgrad-tests
make -C c dylib

TGRAD_LIB="$PWD/.lake/build/lib/libtgrad.dylib" \
  .venv/bin/python -m tgrad bench --shape 64x64x64

The repo is MIT-licensed: github.com/theoriclabs/t-grad.

Closing

The useful thing about Tgrad is not that it is a production ML framework. It is not.

The useful thing is that it makes a pattern concrete:

  • pick a real slice of an existing system;
  • write the done condition as executable gates;
  • keep the authoring surface stable;
  • move semantic/runtime structure into a typed compiled layer;
  • use agents for the grind, but make the harness decide whether the grind succeeded.

That pattern is what I want to carry forward.

Tgrad is now out in public. This post is the marker.
